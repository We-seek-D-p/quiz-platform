# Правила разработки в монорепозитории

## Основные ветки

### `main`

Ветка `main` предназначена только для стабильного кода, готового к деплою.

- не коммитить напрямую в `main`
- не мержить в `main` незавершённую работу
- всё, что попадает в `main`, будет деплоиться через Dokploy

### `develop`

Ветка `develop` — основная интеграционная ветка для разработки.

- все рабочие ветки создаются от `develop`
- все pull request направляются в `develop`
- прямые коммиты в `develop` крайне нежелательны
- предпочтительна работа через отдельные ветки и PR

---

## Именование веток

Название ветки задаётся по схеме:

```text
<service>/<type>/<short-name>
````

### Где:

* `service` — область изменений
* `type` — тип работы
* `short-name` — краткое описание изменений в kebab-case

### Возможные значения `service`

* `auth`
* `management`
* `frontend`
* `session`
* `infra`
* `repo`

### Возможные значения `type`

* `feat` — новая функциональность
* `fix` — исправление ошибки
* `refactor` — рефакторинг без изменения поведения
* `chore` — инфраструктура, конфигурация, техническое обслуживание
* `docs` — документация
* `test` — тесты
* `ci` — CI/CD и автоматизация

### Примеры

```text
auth/fix/jwt-expiration
management/feat/quiz-crud
frontend/refactor/router-guards
infra/chore/nginx-prod-config
repo/ci/selective-docker-build
```

## Формат коммитов

Используется формат **Conventional Commits**:

```text
<type>(<scope>): <message>
```

### Где:

* `type` — тип изменения
* `scope` — область изменений
* `message` — краткое описание действия

### Рекомендуемые `type`

* `feat`
* `fix`
* `refactor`
* `chore`
* `docs`
* `test`
* `ci`
* `build`

### Рекомендуемые `scope`

* `frontend`
* `auth`
* `management`
* `session`
* `infra`
* `repo`

### Примеры

```text
fix(auth): validate refresh token expiration
test(auth): add refresh token expiry tests
feat(frontend): add dashboard layout
fix(auth): refresh token validation
feat(management): implement quiz CRUD
```

---

## Размер и гранулярность коммитов

Коммиты должны быть **небольшими, атомарными и функционально цельными**.

### Основное правило

Один коммит должен отвечать на вопрос:

> Какое одно конкретное изменение было внесено?

Если на коммит нельзя дать короткий и точный ответ, значит он слишком большой.

### Требования

- один коммит = одна логическая единица изменения
- не смешивать в одном коммите:
    - миграции
    - модели
    - репозитории
    - сервисный слой
    - API
    - рефакторинг
    - форматирование
    - инфраструктурные изменения
- если изменение большое, его нужно разбивать на несколько последовательных коммитов
- история должна читаться по коммитам без необходимости открывать весь diff целиком

### Плохо

```text
feat(auth): implement user CRUD
````

Если внутри этого коммита одновременно:

* добавлены модели
* добавлены миграции
* добавлены репозитории
* добавлены сервисы
* добавлены API endpoints
* добавлены схемы и зависимости

Такой коммит слишком большой и плохо ревьюится.

### Хорошо

```text
feat(auth): add user model
chore(auth): add user migration
feat(auth): add user repository
feat(auth): add user service
feat(auth): add user API endpoints
```

или, если это исправление:

```text
fix(auth): validate user email uniqueness
test(auth): add user validation tests
```

### Допустимый подход для большой фичи

Если реализуется большая функциональность, её нужно разбивать на шаги:

1. модель
2. миграция
3. репозиторий
4. сервис
5. API
6. тесты
7. документация

### Форматирование и автофикс

Изменения от линтеров и форматтеров не должны смешиваться с функциональными изменениями.

Например:

* сначала `feat(auth): add user repository`
* отдельно `refactor(auth): apply ruff fixes`

### Цель

Коммит должен быть:

* понятным
* проверяемым
* легко ревьюемым
* при необходимости легко откатываемым

## Breaking changes

Если изменение ломает обратную совместимость, используется `!` после `scope`:

```text
feat(auth)!: change access token payload
```

Такой формат обязателен для изменений, которые:

* меняют публичный API
* меняют контракт данных
* требуют изменений у других сервисов или клиентов

---

## Pull Request

Для любых изменений используется **Pull Request**.

### Правила

* не пушить изменения напрямую в `main`
* рабочие ветки создавать от `develop`
* pull request открывать в `develop`
* один PR должен решать одну понятную задачу
* не смешивать в одном PR инфраструктурные изменения, рефакторинг и бизнес-функциональность без необходимости

### Заголовок PR

Заголовок PR должен соответствовать стилю Conventional Commits.

### Примеры

```text
feat(frontend): add dashboard layout
fix(auth): refresh token validation
feat(management): implement quiz CRUD
```

## Практический процесс работы

1. Переключиться на `develop`
2. Обновить `develop`
3. Создать рабочую ветку по шаблону `<service>/<type>/<short-name>`
4. Выполнить изменения
5. Сделать осмысленные коммиты по Conventional Commits
6. Запушить ветку
7. Открыть Pull Request в `develop`
8. Назначить ревьюера
   - на данный момент ответственным за ревью является `@nikallow`
9. После ревью и проверок смержить PR в `develop`

### Пример

```bash
git checkout develop
git pull --ff-only origin develop
git checkout -b auth/fix/jwt-expiration
```

---

## Дополнительные замечания

* `main` считается продовой веткой
* `develop` — целевая ветка для текущей разработки
* все изменения должны проходить через PR
* история репозитория должна оставаться читаемой и предсказуемой
